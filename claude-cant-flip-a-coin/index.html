Claude Can't Flip a Coin

March 2026

I asked Claude to flip a coin. It said heads. I asked again. Heads. I wrote a script to ask 200 times, each one a completely fresh API call—no memory, no conversation, just the bare prompt and a single-turn response.

200 / 0
heads / tails, across 200 independent API calls

LLMs don't flip coins. They predict the next token. The model isn't choosing between heads and tails—it's answering the question what would a helpful AI assistant most likely say here?

Numbers are worse

I tried the same thing with numbers. "Pick a random number between 1 and 100." Sonnet picks 47. Every time. Opus picks 73. I ran 200 calls and got exactly three distinct values:

200 API calls. "Pick a random number between 1 and 100." That's the whole distribution.

Out of 200 calls, the model produced exactly three distinct values. Different models have different favorites—Opus prefers 73. I don't know why these specific numbers, but they're the same ones humans disproportionately pick when asked to "say a random number." The model isn't generating randomness. It's reproducing the statistical fingerprint of what humans think randomness looks like.

Trying to fix it

I tested ten different prompts, trying to nudge the model toward 50/50. Asking for just H or T instead of the full words. Injecting a random string from Math.random() into each prompt to perturb the attention pattern. A system prompt insisting it be fair. Most of them barely moved the needle.

Then I tried asking for twenty flips at once instead of one.

Fifty API calls, twenty flips each, one thousand total.

502 / 498
heads / tails, across 1,000 flips in batches of 20 (p = 0.69)

Statistically indistinguishable from a fair coin. Five of the sequences:

Heads Tails
Per-sequence P(Heads): mean = 0.502, std = 0.028

But look more carefully. Every sequence has roughly 10 heads and 10 tails. The longest streak in any of them is three. In truly random 20-flip sequences, 77% should contain a run of four or more, and 47% should have a run of five or more. The model produces none.

The standard deviation of the heads ratio across our 50 sequences is 0.028. For true randomness it should be 0.112—four times higher. The sequences are too balanced.

There's a well-known result in psychology: when humans try to write down a "random" sequence, they over-alternate and under-repeat. Streaks feel non-random even though they're statistically expected. So people avoid them. The model learned from those people. It inherited our folk theory of randomness—balanced, streak-averse, neat—rather than randomness itself.

We looked at those sequences and thought they looked random. Of course we did. The model learned what randomness looks like from people, and we can't tell our version apart from the real thing. That's how biases work—you can't see them when you share them.

All experiments used Claude Sonnet 4 (claude-sonnet-4-20250514) at temperature 1.0. Each API call was a single-turn message with no system prompt unless noted. Independence was verified by checking that every response carried a unique message ID. Statistical significance was assessed with a two-tailed binomial test (normal approximation, α = 0.05). Total: ~1,350 API calls.
